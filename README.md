# Only Add Calculator

Это решение тестового задания про калькулятор, который умеет только складывать.
Задача выглядит маленькой, но в требованиях отдельно указаны Clean Architecture, MVP, переиспользуемые assembly и сохранение состояния

Хоть вакансия на Senior Unity Developer, для этого задания явно не подходит движок Unity. Я решил смотреть шире и идти по принципу чистой архитектуры: инструмент должен выбираться под задачу, а не задача под инструмент. Поэтому, несмотря на мой 6-летний опыт разработки match-3 игры на Unity, для этого тестового я разобрался с WPF-приложениями

WPF был выбран сразу: приложение нужно для Windows 11, и логичнее всего использовать .NET

Дальше я продумал архитектуру. Сначала отделил саму логику сложения от вьюшки и файла сохраненного состояния. `Core` (модуль бизнес-логики) не знает ни про WPF, ни про файлы, ни про сообщения пользователю. По ТЗ дроби и отрицательные числа считаются ошибкой, поэтому поддерживаются только неотрицательные целые числа и `+`; вычисления идут через `BigInteger`, чтобы можно было работать с числами любой длины

Следующий модуль - `Application`. В нем находится Presenter и контракты для View и хранилища. View сделана пассивной: WPF-окно только инвоукает события загрузки, нажатия `Result` и закрытия, а Presenter подписывается на них и управляет сценарием

Состояние вынесено в модуль `Persistence`: при закрытии сохраняются текущий ввод и история в `%AppData%\OnlyAddCalculator\state.json`. Если файл отсутствует или поврежден, приложение стартует с пустым состоянием

UI сделан на WPF и не имеет никакой бизнес-логики. Решил для чистоты архитектуры вынести все что связано с текстами проекта в модуль `Localization`. Строки храню в стандартном `Strings.resx`. Можно добавлять локализации других языков без изменения кода и они будут использоваться, если соответствуют локали системы

Чтобы не запутывать все вызовами конструкторов в разных местах я решил добавить DI в проект. После обсуждения с ИИшкой какой DI самый простой и легковесный в .Net я остановился на `Microsoft.Extensions.DependencyInjection`. DI инициализируется в WPF-модуле, потому что это одновременно модуль вьюшки и точка входа в приложение

Для всех модулей кроме вьюшки и локализации написал тесты и небольшой набор интеграционных сценариев без запуска WPF. Сам WPF не тестируется автоматически, потому что основная логика вынесена из окна

P.S. Я не скрываю использование ИИ при выполнении ТЗ. Архитектурные решения принимаю сам, но предварительно обсуждаю их с ИИ: предлагаю вариант, прошу его почелленджить, дорабатываю и фиксирую результат. По итогу создается **исчерпывающий** md-документ в котором описана вся архитектура на верхнем и низком уровне. Этот документ я полностью прочитываю и корректирую. На этот документ уходит бОльшая часть времени т. к. когда он доводится до идеала можно спокойно давать отмашку ИИшке и она не отходя от четкого плана сделает то, что от нее ожидается

При таком флоу я полностью остаюсь в контексте решения, а ИИ не отпускается в свободное плавание и не чудит
